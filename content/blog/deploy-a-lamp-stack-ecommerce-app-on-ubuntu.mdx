### Update the package list

I'll update the package list before installing anything:

```bash
sudo apt update
```

> `sudo apt update` doesn't actually install or upgrade anything — it just refreshes the list of available packages so your system knows what's out there. Think of it as syncing a catalogue before shopping.

## Install the LAMP stack

### Install Apache

```bash
sudo apt install -y apache2
```

Verify it is running:

```bash
sudo systemctl status apache2
```

### Install MariaDB

```bash
sudo apt install -y mariadb-server
```

MariaDB is up and running. Next I'll install PHP and the necessary modules.

### Install PHP

```bash
sudo apt install -y php libapache2-mod-php php-mysql
```

At this point I have the full LAMP stack installed.

## Secure and configure MariaDB

Jump into the MariaDB shell:

```bash
sudo mariadb
```

Execute the following commands to create the database and application user:

```sql
CREATE DATABASE ecomdb;
CREATE USER 'ecomuser'@'localhost' IDENTIFIED BY 'ecompassword';
GRANT ALL PRIVILEGES ON ecomdb.* TO 'ecomuser'@'localhost';
FLUSH PRIVILEGES;
EXIT;
```

### Deploy files and configure Apache

Now I'll copy the application files to Apache's web root:

```bash
sudo cp -r /tmp/learning-app-ecommerce/* /var/www/html/
```

The database connection code uses environment variables rather than hardcoded values. I need to set those variables so Apache can pass them to PHP. I'll do that by editing Apache's environment file:

```bash
sudo nano /etc/apache2/envvars
```

Add these lines at the end of the file:

```bash
export DB_HOST=localhost
export DB_USER=ecomuser
export DB_PASSWORD=ecompassword
export DB_NAME=ecomdb
```

Restart Apache:

```bash
sudo systemctl restart apache2
```

### Fix DirectoryIndex and verify the app

Now I'll test if the app is working:

```bash
curl http://localhost
```

I'm getting the Apache default page, which means Apache is serving `index.html` instead of `index.php`. I need to tell Apache to prioritize `index.php`.

Edit the Apache directory config:

```bash
sudo nano /etc/apache2/mods-enabled/dir.conf
```

You'll see something like:

```
DirectoryIndex index.html index.cgi index.pl index.php index.xhtml index.htm
```

Move `index.php` to the front:

```
DirectoryIndex index.php index.html index.cgi index.pl index.xhtml index.htm
```

Save, then restart Apache:

```bash
sudo systemctl restart apache2
```

Then test again:

```bash
curl http://localhost
```

I can now see the list of products.

## Project deliverable: automate with setup.sh

### Script requirements

Here's what the script needs to do, in order:

1. Update apt packages
2. Install Apache, MariaDB, PHP
3. Start and enable all services
4. Configure the firewall with UFW
5. Configure MariaDB — create database, user, grants
6. Load the SQL data
7. Copy app files to `/var/www/html/`
8. Set environment variables in `/etc/apache2/envvars`
9. Fix `dir.conf` to prioritize `index.php`
10. Restart Apache

### Create and write setup.sh

I'll create the project directory and script:

```bash
mkdir -p /tmp/lamp-ecommerce
nano /tmp/lamp-ecommerce/setup.sh
```

Write the script:

```bash
#!/bin/bash

set -e

echo "==> Updating package list..."
sudo apt update -y

echo "==> Installing Apache, MariaDB, PHP..."
sudo apt install -y apache2 mariadb-server php libapache2-mod-php php-mysql

echo "==> Starting and enabling services..."
sudo systemctl start apache2
sudo systemctl enable apache2
sudo systemctl start mariadb
sudo systemctl enable mariadb

echo "==> Configuring firewall..."
sudo ufw allow OpenSSH
sudo ufw allow 80/tcp
sudo ufw allow 3306/tcp
sudo ufw --force enable

echo "==> Configuring MariaDB..."
sudo mariadb -e "CREATE DATABASE IF NOT EXISTS ecomdb;"
sudo mariadb -e "CREATE USER IF NOT EXISTS 'ecomuser'@'localhost' IDENTIFIED BY 'ecompassword';"
sudo mariadb -e "GRANT ALL PRIVILEGES ON ecomdb.* TO 'ecomuser'@'localhost';"
sudo mariadb -e "FLUSH PRIVILEGES;"

echo "==> Loading inventory data..."
sudo mariadb < /tmp/learning-app-ecommerce/assets/db-load-script.sql

echo "==> Deploying application files..."
sudo cp -r /tmp/learning-app-ecommerce/* /var/www/html/

echo "==> Configuring Apache environment variables..."
echo 'export DB_HOST=localhost' | sudo tee -a /etc/apache2/envvars
echo 'export DB_USER=ecomuser' | sudo tee -a /etc/apache2/envvars
echo 'export DB_PASSWORD=ecompassword' | sudo tee -a /etc/apache2/envvars
echo 'export DB_NAME=ecomdb' | sudo tee -a /etc/apache2/envvars

echo "==> Configuring Apache to serve index.php first..."
sudo sed -i 's/DirectoryIndex index.html/DirectoryIndex index.php index.html/' /etc/apache2/mods-enabled/dir.conf

echo "==> Restarting Apache..."
sudo systemctl restart apache2

echo "==> Done! App should be running at http://localhost"
```

> `set -e` tells the script to exit immediately if any command fails. By default, Bash scripts will keep running the next line even if a previous command returns an error. Adding `set -e` prevents an error from causing a catastrophic chain reaction down the line.

### Configure the firewall with UFW

On a real Ubuntu server — unlike my local Lima VM — the machine is reachable from the network. I need a firewall to block everything by default and only allow the traffic the app actually needs.

**UFW** (Uncomplicated Firewall) is Ubuntu's built-in tool for managing `iptables` rules without writing low-level firewall syntax by hand. The script opens three ports before enabling it:

- **OpenSSH** — keeps SSH access open so I don't lock myself out of the server
- **80/tcp** — allows HTTP traffic to Apache
- **3306/tcp** — allows MariaDB connections, which matters if the database runs on a separate host or another service needs remote access

`ufw --force enable` turns the firewall on non-interactively, which is what I want inside an automated setup script.

### Make the script executable and validate syntax

```bash
chmod +x /tmp/lamp-ecommerce/setup.sh
```

I'll verify the script is valid and has no obvious syntax errors:

```bash

```

> `-n` does a dry run and checks syntax without actually executing anything. If it returns with no output, the script is clean.

## Docker

Before I write anything, I'll think through the architecture. With Docker Compose I'll have two containers talking to each other:

- `web` — Apache + PHP serving the app
- `db` — MariaDB with the `ecomdb` database

The `setup.sh` script was for a single machine where everything runs together. Docker Compose separates the concerns properly, which is also closer to how a real production environment works.

That means I'll need:

- `docker-compose.yml` to define the two services
- `Dockerfile` to build the web container with Apache + PHP + app code
- The SQL file to seed the database on startup

I'll work on the Docker Compose setup on my Mac.

Docker Compose is typically used for local development — I'd run it on my Mac to spin up the app quickly without needing a VM. In production, I'd use something like Kubernetes or ECS to orchestrate containers, not Compose.

So my current workflow looks like this:

As I value the difference between a dev environment and a server environment, I have two deployment approaches: a shell script for provisioning a real Ubuntu server, and a Docker Compose setup for local development. Same app, two different contexts.

- **Lima VM** → simulates a production server, where I deploy manually with `setup.sh`
- **Mac with Docker Compose** → local development environment, where I can spin up the app instantly with one command

Now I'll create the Docker files. I need two things:

- A `Dockerfile` to build the web container
- A `docker-compose.yml` to define both services

I'll also copy the PHP application source into an `app/` directory in the project root — that's what the Dockerfile expects.

```
lamp-ecommerce/
├── app/                  # PHP files from learning-app-ecommerce
├── assets/
│   └── db-load-script.sql
├── docker-compose.yml
├── Dockerfile
├── README.md
└── setup.sh
```

I'll start with the `Dockerfile`. Create it in the project root:

```bash
touch Dockerfile
```

Then open it and add:

```dockerfile
FROM php:8.4-apache

# Install PHP MySQL extension
RUN docker-php-ext-install mysqli

# Copy app files to Apache web root
COPY app/ /var/www/html/

# Generate .env file from environment variables at runtime
CMD ["/bin/sh", "-c", "printf 'DB_HOST=%s\nDB_USER=%s\nDB_PASSWORD=%s\nDB_NAME=%s\n' \"$DB_HOST\" \"$DB_USER\" \"$DB_PASSWORD\" \"$DB_NAME\" > /var/www/html/.env && apache2-foreground"]
```

Then create `docker-compose.yml`:

```yaml
services:
  web:
    build: .
    ports:
      - "80:80"
    environment:
      DB_HOST: db
      DB_USER: ecomuser
      DB_PASSWORD: ecompassword
      DB_NAME: ecomdb
    depends_on:
      - db

  db:
    image: mariadb:11.8
    environment:
      MYSQL_ROOT_PASSWORD: rootpassword
      MYSQL_DATABASE: ecomdb
      MYSQL_USER: ecomuser
      MYSQL_PASSWORD: ecompassword
    volumes:
      - ./assets/db-load-script.sql:/docker-entrypoint-initdb.d/db-load-script.sql
      - db_data:/var/lib/mysql

volumes:
  db_data:
```

Once I've created both files, I'll run:

```bash
docker compose up --build
```

That's the full setup working end to end. Here's a recap of what I have now:

**VM deployment (Lima)**

- The KodeKloud e-commerce app served at http://localhost
- Database seeded automatically on startup
- One command to run the whole thing: `docker compose up`
