<h2 id="introduction">Introduction</h2>

These notes summarize Linux fundamentals that are important for anyone building a career in DevOps, cloud, or infrastructure.

Linux powers modern technology at scale, from the world's fastest supercomputers and most web platforms to the majority of smartphones. In the DevOps ecosystem, many core tools were designed for Linux first: Docker started as Linux-only, Ansible controllers run on Linux, and Kubernetes control planes remain Linux-based.

Because of this, Linux is not just helpful, it is foundational. The material behind these notes uses a story-driven format and hands-on labs to build practical skills in terminal usage, core system concepts, package management, security, networking, storage, and service management with systemd.

### Common challenges

- Trouble with navigating Linux CLI
- Navigating directory structures and files
- No experience working with text editors (VI / Nano)
- Different flavors of Linux
- Errors during installation of apps and dependencies (rpm, apt. dkpg, yum)
- Issues with networking between VMs
- Trouble with permissions and security on Linux
- Lack of hands-on practice

### Learning structures

- Interact with the terminal and understand the shell, which serves as the gateway to the Linux kernel
- Study core Linux concepts that help you understand how systems operate
- Explore various package management methods used by different Linux distributions
- Gain confidence in navigating the command line, manipulating files, and using a variety of text editors
- Understand file-level security, configure permissions, and implement essential Linux security practices
- Troubleshoot networking connectivity issues effectively
- Configure storage options including disk formatting, mounting, and creating logical volume groups
- Set up custom applications to start on boot using SYSTEMD, the primary initializer for Linux services

<h2 id="working-with-shell-part-1">Working with the Shell - Part 1</h2>

### What is a Shell?

The Linux Shell is a program that allows text-based interaction between the user and the OS. This interaction is carried out by typing commands into the interface and receiving the response in the same way.

### Home directory

When logging into the Shell, the very first directory would be the home directory ("/home/michael"). It allows users to store their personal data in the forms of files and directories.

### Command types

- Internal or Built-in Commands: echo, cd, pwd, set...
- External Commands: mv, date, uptime, cp...

To verify whether a command is internal or external, use the type command. For example, executing `type echo` shows that `echo` is a built-in command

### whatis and man commands

- **whatis** offers a brief overview of the command
- **man** provides detailed information including usage examples, options and related data.

### Shell types

- Bourne Shell (sh)
- C Shell (csh or tcsh)
- Korn Shell (ksh)
- Z Shell (zsh)
- Bourne again Shell (bash)

To check the shell being used

```bash
[~]$ echo $SHELL
/bin/bash
```

To change your default shell, use the chsh command.

    ```bash
    [~]$ chsh
    ```

### Bash Shell Features

- Bash auto-completion
- `alias` command

  ```bash
  [~]$ alias dt=date
  ```

- `history` command to view a list of previously executed commands, making it easier to reuse them
- Environment variables

  ```bash
  [~]$ echo $SHELL
  [~]$ env
  ```

- Create a new environment variable

  ```bash
  [~]$ export OFFICE=caleston¨
  ```

- User profile scripts

  User profile scripts, such as `~/.profile`, `~/.bash_profile`, `~/.bashrc`, and others, are executed when a user logs in, allowing the setup of the environment according to personal preferences

  To make changes persistent in Unix-like operating systems, variables, aliases, and other configurations can be added to these profile files.

  ```bash
  echo 'export MY_VARIABLE="example_value"' >> ~/.profile
  ```

  This command appends the `export` statement to the end of the `~/.profile` file, making the variable `MY_VARIABLE` persistent across sessions.

  To add an alias, like ll for ls -l, use:

  ```bash
  echo 'alias ll="ls -l"' >> ~/.profile
  ```

### Customising the Bash Prompt

The Bash prompt, typically seen as [~]$ when you log in, can be customized to display useful system and user information.

The appearance of the Bash prompt is controlled by the `PS1` environment variable. To view its current setting, run:

```bash
[~]$ echo $PS1
[\W]$

# \W = Present Working Directory
# $ = Prompt Symbol
```

To customise the `PS1`

```Bash
[~]$ PS1="[ \d \t \u@\h:\w ] $ "
[Thu Mar 12 22:12:54 bob@caleston:~ ] $
```

The best analogy to understand Linux Kernel is the library analogy. The same interaction model applies: students represent user applications, the librarian represents the kernel, and books/DVDs/workstations represent system resources.

Without librarians, students would search every corner of the library and borrow any materials they find, with no reliable tracking of who borrowed what or when items are due back. In such a mismanaged system, resources might not be available when other students need them most.

A librarian ensures that students get books appropriate to their year level, enforces borrowing limits (for example, one book per student), tracks due dates, sends reminders before deadlines, and makes sure resources are returned.

#### Linux Kernel four major responsibilities

- **Memory Management**: Tracks memory usage and allocation
- **Process Management**: Schedules processes on the CPU and determines their execution order.
- **Device Drivers**: Acts as an intermediary between hardware devices and software processes.
- **System Calls and Security**: Handles requests from processes and enforces security policies.

The Linux Kernel is **monolithic**, which means that the Kernel carries out CPU scheduling, memory management and several other operations by itself.

The Kernel is also **modular**, which means it can extend its capabilities thriugh the use of dynamically loaded Kernel modules

```bash
[~]$ uname
Linux
```

```bash
[~]$ uname -r

```

Where 4 is the Kernel Version, 15 is the Major Version, 0 the Minor Version, 72 the patch release and 'generic' the Distro Specifc Version

#### Understanding Kernel and User Space

Memory management is divided into 2 primary areas:

**Kernel Space**:

- Reserved exclusively for the kernel’s execution and its critical services.
- Processes operating in kernel space have complete access to the system’s hardware resources.
- In our library analogy, kernel space is like a restricted section that only the librarian or administrators can access.

**User Space**:

- Dedicated to processes running outside the kernel, with restricted access to hardware resources.
- This area hosts various utilities, programming languages, and graphical tools collectively known as the userland.
- Similar to the publicly accessible areas of the library designed for readers and reference.

User space programs primarily work with data by performing operations on memory or disk. They communicate their needs through system calls to the kernel.

<h3 id="working-with-hardware">Working With Hardware</h3>

When a USB disk is attached, the kernel device driver detects the state change and emits a **uEvent**. That event is sent to the user-space device manager, **udev**, which then creates the corresponding device node under `/dev` (for example, `/dev/sdb1`).

![The image illustrates the process of hardware interaction, showing a USB disk connecting through device drivers in kernel space and udev in user space to /dev/sdb1.](https://kodekloud.com/kk-media/image/upload/v1752881122/notes-assets/images/Learning-Linux-Basics-Course-Labs-Working-with-Hardware/frame_40.jpg)

Once this flow completes, the new disk appears in the `/dev` filesystem and can be inspected or mounted.

> **Tip:** Linux provides powerful CLI tools to inspect kernel logs, monitor device events, and view hardware configuration details. These are essential for troubleshooting and administration.

#### Viewing Kernel and Device Events

Use `dmesg` to view messages from the kernel ring buffer, including boot-time hardware detection logs. You can pipe output into tools like `less` or filter with `grep` for targeted checks.

Use `udevadm` to query and monitor udev activity. For example, this command returns the udev path for a block device:

```bash
[~]$ udevadm info --query=path --name=/dev/sda5
/devices/pci0000:00/0000:00:17.0/ata3/host2/target2:0:0/2:0:0:0/block/sda/sda5
```

To monitor kernel and udev events in real time:

```bash
[~]$ udevadm monitor
monitor will print the received events for:
UDEV - the event which udev sends out after rule processing
KERNEL - the kernel uevent
```

> **Tip:** `udevadm monitor` is especially useful when attaching or removing devices (for example, unplugging a USB mouse) and validating event flow.

<h3 id="linux-boot-sequence">Linux Boot Sequence</h3>

This section provides a simplified overview of the Linux boot process. The sequence has four key stages:

1. BIOS POST
2. Boot Loader (GRUB2)
3. Kernel Initialization
4. Service Initialization

> **Note:** The Linux boot process starts either when a halted machine powers on or when a running system is rebooted/reset.

![Linux Boot Sequence](/linux-boot-sequence.webp)

#### 1. BIOS POST

The first stage is BIOS POST (Power-On Self-Test). During this phase, firmware validates core hardware components before handing off control. Because POST is firmware-driven, it is mostly independent of Linux itself. If POST fails, the system cannot continue booting.

#### 2. Boot Loader

After POST succeeds, BIOS/firmware reads boot code from the selected boot device (commonly from the disk's early boot sectors and the `/boot` filesystem). The boot loader then displays available boot entries, loads the selected Linux kernel into memory, passes kernel parameters, and transfers execution to the kernel.

In most Linux distributions, this boot loader is **GRUB2** (Grand Unified Boot Loader v2).

#### 3. Kernel Initialization

At this stage, the compressed kernel image is decompressed and loaded into memory. The kernel initializes hardware, memory subsystems, and core drivers, then prepares the system for user space.

Once kernel initialization is complete, Linux launches the init process to bring up user space and essential system processes.

#### 4. Service Initialization

Modern Linux systems typically use **systemd** as PID 1 (init). `systemd` mounts filesystems, starts background services, resolves dependencies, and moves the system to a usable target state.

Compared with legacy SysV init, one major advantage is parallel service startup, which helps reduce boot time.

To confirm which init implementation is linked on your system:

```bash

```

<h3 id="runlevels">Runlevels</h3>

In Linux, the system can operate in several modes, known as runlevels. You might be familiar with graphical mode; however, runlevels allow for various operational modes beyond just the GUI.

To determine the current runlevel of a Linux system, use the `runlevel` command:

```bash theme={null}
Ubuntu 18.04.4 LTS caleston-lp03 tty1
caleston-lp03 login:
```

This output indicates that the system is running at runlevel 3, which is non-graphical mode. In contrast, another laptop could be configured to run at runlevel 5, which provides a graphical user interface.

The boot process utilizes the runlevel setting by having the init process check the current runlevel and start only those services required for that mode. For instance, graphical mode (runlevel 5) needs a display manager service to start the GUI, whereas this service is unnecessary in non-graphical mode.

![The image explains systemd runlevels, showing graphical and command line interfaces, with indicators for display-manager service enabled or disabled.](https://kodekloud.com/kk-media/image/upload/v1752881120/notes-assets/images/Learning-Linux-Basics-Course-Labs-Runlevels/frame_80.jpg)

In modern Linux distributions like Ubuntu 18.04, the system uses systemd as the init process. Under systemd, traditional runlevels are represented as targets:

![The image explains systemd targets and runlevels, showing how they boot into graphical or command line interfaces on RHEL and Ubuntu systems.](https://kodekloud.com/kk-media/image/upload/v1752881121/notes-assets/images/Learning-Linux-Basics-Course-Labs-Runlevels/frame_110.jpg)

In this context:

- Runlevel 5 corresponds to the graphical target.
- Runlevel 3 corresponds to the multi-user target.

While additional runlevels (or targets) exist, these two are the most commonly used.

> If your system boots into a command line interface by default, it's likely configured to use the multi-user target.

#### Changing the Default Target

To view the current default systemd target, execute the following command, which reads the file located at `/etc/systemd/system/default.target`:

```bash theme={null}
$ systemctl get-default
graphical.target

$ ls -ltr /etc/systemd/system/default.target
```

In this example, the `default.target` file is a symbolic link to the `graphical.target` unit file located in the `/lib/systemd/system/` directory.

If you wish to change the default target, use the `systemctl set-default` command followed by your desired target. For instance, to switch from graphical mode (runlevel 5) to multi-user mode (runlevel 3), run:

```bash theme={null}
$ systemctl get-default
graphical.target

$ ls -ltr /etc/systemd/system/default.target
/etc/systemd/system/default.target -> /lib/systemd/system/graphical.target

$ systemctl set-default multi-user.target
Created symlink /etc/systemd/system/default.target -> /lib/systemd/system/multi-user.target
```

This command updates the default target, effectively switching your system's mode of operation from graphical to non-graphical.

> After changing the default target, you may need to reboot your system for the changes to take effect.

<h3 id="file-types">File Types</h3>

| File Type     | Sub File Type      | Description                                                                                                                            |
| ------------- | ------------------ | -------------------------------------------------------------------------------------------------------------------------------------- |
| Regular File  | -                  | Common files and contain text, data, images, etc. Examples include configuration files, shell scripts, and JPEG images.                |
| Directory     | -                  | Contains references to files and subdirectories.                                                                                       |
| Special Files | Character Files    | usually located in the /dev directory, represent devices that interact serially with the operating system (e.g., a mouse or keyboard). |
| -             | Block Files        | Also found under /dev, block files represent devices that read and write data in fixed-size blocks, such as hard disks and RAM.        |
| -             | Links (Hard Links) | A hard link associates multiple filenames with the same block of data. Deleting one hard link can result in the removal of that data.  |

| -             | Sockets            | Sockets are special files that facilitate communication between different processes.                                                   |
| -             | Named Pipes        | Also known as FIFOs, these files allow one process to deliver its output directly to another using a unidirectional data flow.         |

```bash
[~]$ file /home/michael/

```

```bash

```

| Symbol | File Type |
| --- | --- |

| s | Socket |
| p | Named pipe |

| `/home`             | Stores home directories for regular users.       | The root user's home is `/root`, not under `/home`.                  |
| `/opt`              | Installation location for third-party software.  | Good place for custom app deployments.                               |
| `/mnt`              | Temporary mount point for filesystems.           | Commonly used to mount network shares during setup tasks.            |
| `/tmp`              | Stores temporary files and transient data.       | Copy installer packages here during setup, then clean up after use.  |

| `/usr`              | Contains userland applications and related data. | Hosts software such as editors and browsers on modern systems.       |
| `/var`              | Stores variable data such as logs and caches.    | First place to check during troubleshooting and incident diagnosis.  |

<h2 id="linux-package-management">Linux Package Management</h2>

<h3 id="package-management-introduction">Introduction</h3>

A Linux distribution (or "distro") is a complete operating system built by bundling the core Linux kernel with various software, tools, and a package manager to make it ready for use.

#### What makes up a Distro?

Think of a distribution as a "pre-packaged meal" where the Linux kernel is the main ingredient, but each chef (the distro creator) adds different sides and seasonings:

- Linux Kernel: The core engine that talks to your hardware.
- Package Manager: The tool used to install and update software (like the App Store).
- Desktop Environment: The visual interface (windows, taskbars, and menus).
- Default Apps: Pre-installed software like a web browser (Firefox) or office suite (LibreOffice).

#### Debian-based vs. Red Hat-based

Most Linux systems today belong to one of these two "families." They differ primarily in philosophy, package management, and who they are built for

| Feature         | Debian-based                                                     | Red Hat-based                                                      |
| --------------- | ---------------------------------------------------------------- | ------------------------------------------------------------------ |
| Philosophy      | Strictly community-driven; focuses on free open-source software. | Commercial focus; built for enterprise stability and paid support. |
| Package Format  | Uses `.deb` files.                                               | Uses `.rpm` files.                                                 |
| Tools           | APT (high-level) and `dpkg` (low-level).                         | DNF or YUM (high-level) and RPM (low-level).                       |
| Popular Distros | Ubuntu, Linux Mint, Kali Linux, Pop!\_OS.                        | Fedora, RHEL, Rocky Linux, AlmaLinux.                              |
| Support         | Community-led (forums, wikis, and volunteers).                   | Professional (paid 24/7 corporate support from Red Hat).           |

<h3 id="rpm-and-yum">RPM and YUM</h3>

RPM and YUM are the primary package managers for RPM-based Linux distributions. Both tools play essential roles in managing software packages by handling installation, upgrades, removal, and dependency resolution.

#### RPM (Red Hat Package Manager)

| Mode         | Command                                                                                                                                               |
| ------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| Installation | `rpm -ivh telnet.rpm`<br /><br />`-i`: Install the package<br />`-v`: Verbose mode for detailed output.<br />`-h`: Display the installation progress. |
| Uninstalling | `rpm -e telnet.rpm`                                                                                                                                   |
| Updgrade     | `rpm -Uvh telnet.rpm`<br /><br />Note the uppercase `U`                                                                                               |
| Query        | `rpm -q telnet.rpm`                                                                                                                                   |
| Verifying    | `rpm -Vf <path to file>`<br /><br />Note the uppercase `V`                                                                                            |

> One important limitation of **RPM** is that it does not automatically resolve package dependencies. This shortcoming is addressed by a higher-level package manager known as **YUM**

#### YUM (Yellowdog Updater Modified)

Repository configurations are stored in `.repo` files located in the `/etc/yum.repos.d` directory. YUM leverages these configuration files to locate and manage software packages. Repositories can be local or remote.

##### YUM Installation Process

When you run a yum install command, YUM conducts a comprehensive transaction check:

- It verifies if the package is already installed.
- It searches the enabled repositories in /etc/yum.repos.d for the package.
- It checks for any missing or outdated dependencies.

##### Common YUM Commands

| Task                      | Command             | Use Case                                                     |
| ------------------------- | ------------------- | ------------------------------------------------------------ |
| List repositories         | `yum repolist`      | Display all configured repositories                          |
| Identify package source   | `yum provides scp`  | Find which package provides a specific command (e.g., `scp`) |
| Remove a package          | `yum remove httpd`  | Uninstall a package from the system                          |
| Update a specific package | `yum update telnet` | Upgrade a package (e.g., Telnet)                             |
| Update all packages       | `yum update`        | Upgrade all installed packages                               |

<h3 id="dpkg-and-apt">DPKG and APT</h3>

DPKG and APT are package managers in Debian-based distributions.

#### DPKG (Debian Package Manager)

| Task                      | Command                  |
| ------------------------- | ------------------------ |
| Install or update package | `dpkg -i telnet.deb`     |
| Remove package            | `dpkg -r telnet`         |
| List package details      | `dpkg -l telnet`         |
| Check package status      | `dpkg -s telnet`         |
| Verify the package        | `dpkg -p <path to file>` |

#### APT (Advanced Package Tool)

APT (Advanced Package Tool) is a user-friendly front-end for DPKG, which means it acts as a "manager" or "translator" to make a package management easier.

APT relies on software repositories defined in the sources file. The repository details are stored in:

```bash
/etc/apt/sources.list
```

**Common APT Commands**

Below is a table summarizing key APT commands along with their use cases:

| --- | --- | --- |

| Remove a package | `apt remove telnet` | Remove an installed package |
| Search for a package | `apt search telnet` | Search for a package within the repository |

#### Checking the size of a file

| Task                                                                         | Command                | Output             |
| ---------------------------------------------------------------------------- | ---------------------- | ------------------ |
| To view files sizes in kilobytes                                             | `[~]$ du -sk test.img` | `100000`           |
| For a more human-readable format                                             | `[~]$ du -sh test.img` | `98M     test.img` |
| Use the `ls -lh` command to view<br /> file sizes in a human-readable format | `[~]$ ls -lh test.img` | -                  |

#### Using `tar` for archival

The `tar` command is one of the most common utilities used to group multiple files or directories into a single archive file (**tarball**). This is particularly useful for data backup, transfer, and archiving.

To create a tar archive, use the `-c` option to create the archive and the `-f` option to specify the file name:

```bash
[~]$ tar -cf test.tar file1 file2 file3
[~]$ ls -ltr test.tar
-rw-rw-r--  1 1281054720 Mar 13 19:48 test.tar
```

You can list the contents of a tarball using:

```bash
[~]$ tar -tf test.tar
./file1
./file2
./file3
```

To extract files from a tar archive, use the `-x` option:

```bash
[~]$ tar -xf test.tar
```

Additionally, to compress the tarball and reduce its file size, include the `-z` option:

```bash
[~]$ tar -zcf test.tar file1 file2 file3
```

#### Compression commands

Linux offers several commands for compressing files. The choice of command depends on the type of data and required compression level. Below are examples using `bzip2`, `gzip`, and `xz`:

| Utility | Compressing           | File extension | Uncompressing               |
| ------- | --------------------- | -------------- | --------------------------- |
| `bzip2` | `[~]$ bzip2 test.img` | `.bz2`         | `[~]$ bunzip2 test.img.bz2` |
| `gzip`  | `[~]$ gzip test1.img` | `.gz`          | `[~]$ gunzip test1.img.gz`  |
| `xz`    | `[~]$ xz test2.img`   | `.xz`          | `[~]$ unxz test2.img.xz`    |

> It is important to note that you do not always need to fully uncompress a file to view its content. Tools like `zcat`, `bzcat`, and `xzcat` allow you to read compressed files without manual decompression.

<h3 id="files-and-patterns">Searching for Files and Patterns</h3>

#### Using the `locate` command

The `locate` command provides a fast way to find files by querying a pre-built database (mlocate.db). For example, to search for files named “City.txt” within a directory structure representing continents, countries, and cities, simply run:

```bash
locate City.txt
/home/michael/Africa/Egypt/Cairo/City.txt
/home/michael/Asia/India/Mumbai/City.txt
```

> If the file isn’t listed, you may need to update the database manually by running `sudo updatedb`, then re-run the `locate` command

#### Using the `find` command

Unlike `locate`, the `find` command does not rely on a pre-built database and offers a highly versatile suite of options for searching. To locate a file named “City.txt” within the directory /home/michael, use:

```bash
find /home/michael -name City.txt
/home/michael/Africa/Egypt/Cairo/City.txt
/home/michael/Asia/India/Mumbai/City.txt
```

#### Using the `grep` command

Assume you have a file named sample.txt with the following content:

```bash
[~]$ cat sample.txt
This is the first line.
Followed by the second line.
And then the third line.
The fourth line has CAPITAL LETTERS
The fifth line does not want to be printed
```

| Command                                         | Description                                                                                |
| ----------------------------------------------- | ------------------------------------------------------------------------------------------ |
| `grep second sample.txt`                        | prints only the line that contains the word “second.”                                      |
| `grep -i capital sample.txt`                    | for a case-insensitive search, add the `-i` flag                                           |
| `grep -R "third line" /home/michael`            | search recursively through a directory<br /> with the `-R`flag                             |
| `grep -v exam examples.txt`                     | print only the lines that do NOT match<br /> a particular pattern with the `-v` flag       |
| `grep -w exam examples.txt`                     | match only the whole word “exam”<br /> with the `-w` option                                |
| `grep -A1 Arsenal premier-league-table.txt`     | print the line containing “Arsenal” along with<br /> the line immediately **following** it |
| `grep -B1 Arsenal premier-league-table.txt`     | print the line containing “Arsenal” along with<br /> the line immediately **preceding** it |
| `grep -A1 -B1 Arsenal premier-league-table.txt` | combine both options to display one line<br /> before and one after a matching line        |

<h3 id="io-redirection">IO Redirection</h3>

This section delves into IO redirection in Linux and explain the concept of standard streams

Every Linux command you run is automatically associated with three primary data streams:

- **STDIN**: The standard input stream that accepts text input.
- **STDOUT**: The standard output stream that displays text output on your screen.
- **STDERR**: The standard error stream that shows error messages.

Below is an example of displaying the content of a file using the cat command:

```bash
[~]$ cat sample_text.txt
This is the file contents
```

If you attempt to access a file that does not exist, the error message is directed to **STDERR**:

```bash
[~]$ cat sample_text.txt
This is the file contents
cat: sample_text.txt: No such file or directory
```

#### Redirecting Standard Output and Error

To redirect the standard output to a file instead of the screen, use the forward arrow symbol (`>`).

Use a **single** arrow symbol to **overwrite** the file:

```bash
[~]$ echo $SHELL > shell.txt
[~]$ cat shell.txt
/bin/bash
```

Use a **double** arrow symbol to **append** to an existing file:

```bash
[~]$ echo "This is the Bash shell" >> shell.txt
[~]$ cat shell.txt
/bin/bash
This is the Bash shell
```

To redirect only error messages to a file, prefix the redirection operator with the number `2`. This tells the shell to redirect **STDERR**:

```bash
[~]$ cat missing_file 2> error.txt
[~]$ cat error.txt
cat: missing_file: No such file or directory

# append error msg instead of overwritting
[~]$ cat missing_file 2>> error.txt
```

#### Using Pipes to Link Commands

Often, you might want to pass the output of one command directly to another without creating an intermediary file—that’s where piping comes into play.

The pipe operator (|) transfers the standard output from the command on its left to the standard input of the command on its right. For example:

```bash
command1 | command2
```

The pipe mechanism allows you to chain multiple commands together, enhancing efficiency and flexibility.

#### The `tee` command

Another valuable tool for managing IO redirection is the tee command. Unlike the redirection operator that sends the output exclusively to a file, tee duplicates the output by writing it both to a file and to the screen simultaneously.

```bash
[~]$ echo $SHELL | tee shell.txt
/bin/bash
[~]$ cat shell.txt
/bin/bash
```

To append the output to the file rather than replacing its content, use the `-a` option with tee:

```bash
[~]$ echo $SHELL | tee -a shell.txt
```

By mastering these IO redirection techniques, you can efficiently manage command outputs and errors in Linux. Whether saving output to files, suppressing error messages, or chaining commands with pipes, these methods are essential for streamlining your workflow and enhancing your command-line proficiency.

<h3 id="vi-editor">VI Editor</h3>

> A guide to using the VI Editor in Linux, covering its modes, commands, and features for efficient text editing and coding.

Explore the power of console-based text editors in Linux with a focus on the VI Editor. While the simple `cat` command is useful for small file operations, its limited functionality makes it impractical for handling extensive text manipulation or coding. This guide will walk you through using VI, one of the most robust editors available by default on nearly every Linux distribution.

#### Getting Started with VI Editor

The VI Editor is a powerful, console-based text editor perfect for editing files and writing code. To open a file for editing, simply invoke VI followed by the file name. For instance, to open a sample file from Michael's home directory, use the following command:

```bash theme={null}
[$] vi /home/michael/sample.txt
This is the first line.
Followed by the second line.
Third line is very long compared to the previous two lines.
Hello there!
hello there!
~
~
~
~
~
~
~
~
~
~
~
"sample.txt" 5L, 139C
1,1 All
```

When you open a file with VI, it starts in Command Mode and displays the file's existing content (if any). VI Editor operates primarily in three modes:

1. **Command Mode** - Execute commands such as copying, pasting, deleting, and navigating.
2. **Insert Mode** - Edit or add new text.
3. **Last Line Mode** - Save, quit, or perform other file operations.

#### Opening Files in Command Mode

Every time you start VI, it opens in Command Mode. In this state, only commands are recognized. Navigate through the file using the arrow keys or by pressing `H` (left), `J` (down), `K` (up), and `L` (right). You may also use the mouse to select text.

#### Switching to Insert Mode

To make changes in your file, switch from Command Mode to Insert Mode by pressing lowercase `i`. In Insert Mode, a blinking cursor and the indicator `-- INSERT --` appear at the bottom of the screen, allowing you to add or modify text. Other keys such as `a`, `o`, `I`, `A`, or `O` may also be used to enter Insert Mode based on where you want to position the cursor.

> **Tip:** To exit Insert Mode and return to Command Mode, simply press the `Esc` key.

#### Working with VI Commands

Once back in Command Mode, you can perform a variety of operations:

- **Copying Text:** Place the cursor on a line and press `YY` to copy it. To paste the copied line above the target line, move the cursor appropriately and press `P`.

  ```bash theme={null}
  This is the first line.
  Followed by the second line.
  Third line is very long compared to the previous two lines.
  Hello there!
  hello there!
  ~
  ~
  ~
  "sample.txt" 5L, 139C 1,1 All
  ```

- **Saving and Deleting:**
  - Save the file using uppercase `ZZ`.
  - Delete a character by positioning on it and pressing `X`.
  - Delete an entire line with `DD`.
  - For multiple lines, use a command like `D3D` to delete three lines (replace `3` with any number).

  Here's an example of deletion in Command Mode:


  - Press `U` to undo an unintended change.
  - Use `Control-R` to redo a change that was undone.

  ```bash theme={null}
  This is the first line.
  Followed by the second line.

  Hello there!
  hello there!
  ~
  ~

  ~
  ~

  ```
