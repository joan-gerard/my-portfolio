When I started studying networking for DevOps, I expected it to be mostly memorizing commands. It turned out to be more about building a mental model of how traffic moves.

Once that model clicked, commands like `ip addr`, `ip route`, and `nslookup` stopped feeling like random syntax and started feeling like tools for answering one question:

**Where is this packet trying to go, and who decides the next step?**

This chapter covers the foundations that made networking feel practical: switching, routing, gateways, IP addressing, and DNS on Linux.

<h2 id="the-big-picture">The Big Picture: How Machines Talk</h2>

At a high level, networking comes down to four core ideas:

- **Switching**: communication inside the same network
- **Routing**: communication between different networks
- **Gateway**: the default exit path out of your local network...used when no more specific route exists in the routing table
- **IP addressing**: how devices are identified

If you can reason about those four, most day-to-day troubleshooting becomes manageable.

<h2 id="traffic-flow">Traffic Flow: Same Network vs Different Network</h2>

### 1) Same network: switching

Before traffic is sent, ARP is used to map the destination IP to a MAC address. If two machines are on the same subnet (for example `192.168.1.10` and `192.168.1.11`), they can communicate through a switch.

Each system needs:

1. A network interface (such as `eth0`)
2. An assigned IP address

You can validate connectivity with:

```bash
ping 192.168.1.11
```

`ping` uses ICMP to test reachability and basic network connectivity.

### 2) Different networks: routing

A switch cannot move traffic between separate networks such as:

- `192.168.1.0/24`
- `192.168.2.0/24`

That requires a router. The router is the device that interconnects networks and forwards packets between them.

### 3) How a host decides: routing table

Every Linux host keeps a routing table. This is the lookup map used to decide where outgoing traffic should be sent.

```bash
ip route
```

### 4) The gateway: your network's exit door

The default gateway is the path used when traffic is destined outside the local network.

Practical flow:

1. Laptop to printer (same subnet): local switching path
2. Laptop to `google.com`: not local, so send to default gateway
3. Gateway/router forwards traffic outward toward internet routes

In many home networks, the gateway is often something like `192.168.1.1` or `10.0.0.1`.

<h2 id="linux-networking-commands-worth-memorizing">
  Linux Networking Commands Worth Memorizing
</h2>

These commands appeared repeatedly in labs and are useful in real operations:

```bash
ip link
ip addr  # or ip a (common shorthand)
ip addr add 192.168.1.10/24 dev eth0
ping 192.168.1.11
ip route
ip route add ...
cat /proc/sys/net/ipv4/ip_forward
echo 1 > /proc/sys/net/ipv4/ip_forward
```

Notes:

- `ip addr add ...` is temporary unless persisted in system config.
- `ip_forward` This enables the host to forward packets between interfaces, effectively allowing it to act as a router..
- Permanent forwarding is typically configured via `sysctl` settings.

<h2 id="concepts-that-helped-me">Concepts That Helped Me the Most</h2>

### Network interface and NIC

- A **network interface** is the host's connection point to a network (physical or virtual).
- A **NIC** (Network Interface Card) is the physical hardware component enabling that connectivity.

No interface means no network communication.

### Switch vs router

- **Switch**: connects devices inside one network; forwards using MAC addresses (OSI Layer 2).
- **Router**: connects different networks; forwards using IP addresses (OSI Layer 3).

Routers in home environments also commonly perform NAT to share one public ISP IP across internal devices.

### Home network reality

A typical home setup is one network, but multiple networks commonly appear when you add:

- Guest Wi-Fi (isolated subnet)
- IoT segmentation (often VLAN-based)
- Double NAT (router behind router)

Those patterns explain many "why can't this device reach that one?" problems.

<h2 id="turning-names-into-reachable-ips">
  DNS: Turning Names into Reachable IPs
</h2>

After learning packet flow, DNS made more sense because it answers a different question: **what IP should this name map to?**

### Local name resolution with `/etc/hosts`

If host B is `192.168.1.11`, host A can map a friendly name locally:

```bash
echo "192.168.1.11 db" | sudo tee -a /etc/hosts
```

Then:

```bash
ping db
ssh db
```

Key behavior:

- `/etc/hosts` is local and authoritative for that machine
- It does not verify remote hostname identity
- You can map multiple names to one IP

Example:

```bash
echo "192.168.1.11 www.google.com" | sudo tee -a /etc/hosts
```

This forces local resolution of that name to the specified IP.

<h2 id="central-dns">Why Central DNS Exists</h2>

Managing many hosts with local `/etc/hosts` files does not scale. If one server IP changes, every client file has to be updated manually.

A DNS server centralizes those mappings.

Configure resolver settings in:

```bash
echo "nameserver 192.168.1.100" | sudo tee -a /etc/resolv.conf
```

> On many modern systems, /etc/resolv.conf is managed automatically, so changes may not persist.

Resolution order is controlled by `/etc/nsswitch.conf`, commonly:

```text
hosts: files dns
```

That means: check local files first, then query DNS.

<h2 id="dns-hierarchy">DNS Hierarchy in Plain Terms</h2>

For names like `www.google.com`, resolution may traverse:

1. Root DNS servers
2. TLD servers (for `.com`)
3. Authoritative servers for `google.com`

Organisations follow the same model with their own domains and subdomains (`mail.mycompany.com`, `hr.mycompany.com`, and so on).

<h2 id="common-dns-record-types">Common DNS Record Types</h2>

- **A**: hostname to IPv4 address
- **AAAA**: hostname to IPv6 address
- **CNAME**: alias name pointing to another hostname

These are the core records you will constantly encounter.

<h2 id="dbs-diagnostics">
  DNS Diagnostics: <code>nslookup</code> and <code>dig</code>
</h2>

`ping` can confirm resolution in context, but dedicated DNS tools give deeper visibility:

```bash
nslookup www.google.com
dig www.google.com
```

Important caveat from the labs:

- `nslookup` and `dig` query DNS servers directly, which helps isolate DNS issues from local resolution (such as /etc/hosts).
- This is useful when you need to separate "local override issue" from "real DNS issue."

## Search Domains: Using Short Names

You can reduce typing by configuring a search domain:

```bash
echo "search mycompany.com" | sudo tee -a /etc/resolv.conf
```

Then short names (for example `app01`) can resolve as `app01.mycompany.com`.

This is convenient in internal environments where most lookups belong to one domain.

<h2 id="lab-safe-command-pattern">Lab-Safe Command Pattern (<code>tee</code> vs direct redirection)</h2>

A useful pattern from this chapter:

```bash
echo "127.0.0.1 www.google.com" | sudo tee -a /etc/hosts
```

Why this matters:

- `sudo` applies to `tee` (the writer), so protected files can be updated safely.
- `-a` appends instead of overwriting.
- Avoids permission issues common with `sudo ... >> file` redirection patterns.

<h2 id="qa">Practical Q&A (Condensed)</h2>

- **Can a system have multiple IPs?** Yes, across multiple interfaces or multiple addresses on one interface.
- **Which Linux command shows kernel routing table?** `route` (or `ip route`).
- **If local hosts and DNS disagree, which wins by default?** `/etc/hosts` when `hosts: files dns`.
- **Top-level domain in `www.google.com`?** `.com`.
- **Example subdomain?** `maps.google.com`.
- **Tool to query hostname from DNS server?** `nslookup` (or `dig` for deeper details).

<h2 id="final-reflection">Final Reflection</h2>

Networking started to make sense once I stopped memorizing commands and focused on flow.

Every issue now maps back to the same chain: how a system identifies itself, how it reaches other networks, and how names resolve to addresses. When something breaks, the problem is almost always somewhere along that path.

For DevOps work, this mental model matters more than any individual command. It is what turns debugging from guesswork into a structured process.
